Definition

What is photogrammetry?

photo (light) + gram (drawing) + metry (measurement)

The science of obtaining reliable measurements and 3D shape of real objects from overlapping 2D photographs taken from different viewpoints.

Your eyes already do this — two viewpoints (left + right eye) let your brain triangulate depth. Photogrammetry does it with many cameras and math.

The core idea

How do 2D pixels become 3D?

  • Overlap is everything. The same surface point must appear in several photos.
  • Find matching features across those photos (corners, textures, keypoints).
  • Solve two things at once — where each camera was, and where each 3D point is.
  • Triangulate matched rays to recover depth, then densify into a full surface.

The golden rule

More overlapping viewpoints = more matched rays = more accurate geometry.

This is exactly why our app asks the user to capture the object from multiple angles before it reconstructs.

Classic pipeline

Structure-from-Motion → Mesh

1

Feature Detection

Find distinctive keypoints (SIFT/ORB) in every image.

2

Feature Matching

Match the same points across overlapping photos.

3

Structure-from-Motion

Recover camera poses + a sparse 3D point cloud.

4

Bundle Adjustment

Globally optimize to minimize re-projection error.

5

Dense / MVS

Multi-View Stereo fills in a dense point cloud.

6

Meshing

Points → triangle surface (e.g. Poisson).

7

Texturing

Project original photos back onto the mesh.

8

Export

Output OBJ / glTF / GLB for viewing & AR.

✦

Result

A measurable, textured 3D replica of reality.

Two roads to 3D

Classical photogrammetry vs AI reconstruction

🧭 Classical (SfM / MVS)

  • Needs many well-overlapped photos (30–100+)
  • Geometrically accurate & measurable
  • Struggles with shiny / textureless / thin parts
  • Heavy compute; minutes to hours
  • Tools: RealityCapture, Meshroom, Metashape

🧠 AI / Generative (our app)

  • Works from a few images (we send 4)
  • Neural priors hallucinate unseen surfaces
  • Robust to sparse views; fast turnaround
  • Runs in the cloud — light on the device
  • Tools: Meshy AI, Luma, Tripo, NeRF/Gaussian Splatting

💡 We use the AI approach: it trades survey-grade precision for speed and a great mobile experience — ideal for asset visualization.

Choosing wisely

So when do you pick what?

✦ Reach for Meshy when…

  • You have few photos (or just text) and need a model now
  • You want a clean, game-ready asset, not a survey
  • You need an API to embed in an app — like we did
  • Missing sides can be plausibly invented

Reach for the others when…

  • Metashape / RealityCapture → survey-grade accuracy, drone mapping, dimensions matter
  • Meshroom → free, offline, full control of the pipeline
  • Polycam / KIRI → quick phone scans, LiDAR rooms & objects
  • Luma AI → cinematic scenes & Gaussian-splat flythroughs

For EAM asset visualization on a phone, Meshy's speed + low input + API win — accuracy isn't the priority, turnaround and UX are.

Step 1 · Photogrammetry starts here

Capture: multiple viewpoints 📷

  • Live preview with CameraX bound to the Compose lifecycle.
  • Runtime camera permission handled before preview binds.
  • A counter enforces the required number of shots — each is a new viewpoint.
  • Thumbnails give instant feedback so the user covers all angles.
This is the data-acquisition stage of photogrammetry — the quality of these shots decides the quality of the model.
// CameraScanScreen.kt — take a photo
imageCapture.takePicture(outputOptions, executor,
  object : OnImageSavedCallback {
    override fun onImageSaved(r) {
      // crop to a centered object frame
      val uri = ImageProcessor
        .centerCropToObjectFrame(context, file)
      onImageCaptured(uri)
    }
  })
Step 2 · Clean the input

Preprocess: focus on the object ✂️

  • Center-crop to a square, keeping the subject framed and removing distracting background.
  • Re-encode to JPEG at 92% quality — small enough to upload, sharp enough to reconstruct.
  • Bitmaps are recycled and temp files cleaned up to protect memory.

Consistent framing helps the AI focus on the target — the app's nod to the classic "isolate your subject" rule.

// ImageProcessor.kt
val size = min(bmp.width, bmp.height)
val left = (bmp.width  - size) / 2
val top  = (bmp.height - size) / 2
val cropped = Bitmap.createBitmap(
    bmp, left, top, size, size)
cropped.compress(JPEG, 92, stream)
Step 3 · Send to the engine

Encode & upload ☁️

  • Each image → Base64 data URI (data:image/jpeg;base64,…).
  • Payload capped at 4 images to match the API limit.
  • Sent as JSON with texturing + PBR flags and target format glb.
  • Runs on Dispatchers.IO, wrapped in runCatching → Result.
// MeshyRepository.kt
val dataUris = uris.take(4).map { it.toDataUri() }
api.createTask(
  CreateTaskRequest(
    imageUrls    = dataUris,
    shouldTexture = true,
    enablePbr     = true,
    targetFormats = listOf("glb")),
  apiKey = "Bearer $KEY")

🔐 The API key is injected at build time via BuildConfig — kept out of source control.

Step 4 · The reconstruction job

An asynchronous cloud task

3D reconstruction takes time, so Meshy is a job queue, not a blocking call:

  • POST returns a task id immediately.
  • We persist that id to local history (SharedPreferences + Gson).
  • GET /{id} reports status, progress and, when ready, model_urls.
  • The user can leave and come back — the id survives app restarts.
// MeshyApi.kt
@POST("openapi/v1/multi-image-to-3d")
suspend fun createTask(...): CreateTaskResponse

@GET("openapi/v1/multi-image-to-3d/{id}")
suspend fun getTask(
  @Path("id") id: String, ...
): MeshyTaskResponse
// → status, progress, model_urls[glb]
Step 5 · Retrieve the asset

Download the GLB 📦

  • When model_urls["glb"] exists, we stream the file with OkHttp.
  • Saved via MediaStore into Downloads/Meshy_EAM/ (scoped storage, Android 10+).
  • Cached locally — re-opening a model skips the network entirely.
GLB = binary glTF: geometry + textures + materials in one portable file. The "JPEG of 3D."
// getOrDownloadModelFilePath()

Step 6 · See it in 3D

Render the model 🧊

  • SceneView (Google Filament PBR engine) renders the GLB in Compose.
  • A ModelNode is scaled to unit size and centered automatically.
  • Camera node positioned back on Z so the whole object is in frame.
  • Loading + error states handled while the model instance is created.

Field tips

How to capture good photos

🔄

Orbit the object

Move around it; don't just rotate on the spot. Cover every side.

🧩

~60–80% overlap

Each photo should share most of its content with the next.

💡

Flat, even light

Avoid hard shadows and harsh highlights that confuse matching.

🎯

Sharp & in focus

Blur destroys features. Keep the object filling the frame.

🚫

Avoid shiny/clear

Mirrors, glass and glossy metal break reconstruction.

🌫️

Texture helps

Matte, detailed surfaces reconstruct far better than blank ones.

Reality check

Limits & challenges

Physics is hard

  • Reflective / transparent / textureless surfaces
  • Thin structures & fine detail (wires, hair)
  • Moving subjects and changing light
  • Scale ambiguity without a reference / GPS

In our app specifically

  • Only 4 images → AI infers a lot; less precise
  • Depends on cloud latency & network
  • Base64 inflates payload size (~33%)
  • No automatic polling loop yet — user re-checks history
